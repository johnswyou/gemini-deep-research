## Codebase Patterns

Read this section BEFORE editing. Rules discovered the hard way — don't

### Tooling / lint
- **Ruff bugbear B008:** Typer's `typer.Option(...)` / `typer.Argument(...)`
  in default values trip B008. Fixed via
  `[tool.ruff.lint.flake8-bugbear] extend-immutable-calls = ["typer.Option", "typer.Argument"]`
  in `pyproject.toml`. DO NOT add `# noqa: B008` — use this config.
- **Ruff PLC0415 (import not at top):** only two exceptions are allowed
  project-wide. (a) `from google import genai` is lazy-imported inside
  `GdrClient.__init__` so `gdr --help` / `--version` stay fast. Use
  `# noqa: PLC0415` with a comment. (b) `src/gdr/cli.py` has a per-file
  ignore for PLC0415 (declared in `pyproject.toml`).
- **Ruff RUF001/RUF003 (ambiguous unicode):** en-dash `–` trips in strings
  and comments. Use ASCII hyphen `-`. Em-dash in docstrings is fine per
  existing code (they're not flagged).
- **Ruff RUF100 (unused noqa):** `A002` and `BLE001` are NOT in the lint
  set — don't add noqa for them.
- **Ruff PLR0915 (too many statements):** the 50-statement limit catches
  `run()` easily. Extract helpers like `_finalize_and_render`,
  `_consume_create_result`, `_safe_build_client`.
- **Ruff PLW2901 (for-loop var reassignment):** rename `stripped = line.strip()`
  to avoid reassigning loop var `line`.
- **Ruff RUF059 (unused unpack):** prefix unused tuple members with `_`:
  `agg, _events = _collect(...)`.
- **Ruff E402 (imports not at top):** don't put module-level assignments
  (like `_UTC = timezone.utc`) between imports. Put them after ALL imports.

### mypy (strict, target py310)
- `datetime.UTC` is 3.11+. Use `from datetime import timezone` and
  `timezone.utc` (aliased as `_UTC = timezone.utc` at module scope, after
  imports). This is project-wide.
- `tomli` import has a mypy override (only imported on 3.10 branch) in
  pyproject.
- google.genai has a mypy override (ignore_missing_imports).
- Function naming collision: `Store.list` shadows the builtin `list` in
  class scope, breaking `list[Record]` annotations. This class uses
  `Store.recent(...)` instead. DO NOT rename back.
- Cast `policy.redact(raw)` to `dict[str, Any]` — the redactor returns
  `Any` generically.

### Domain architecture
- **`execute_research()` is the shared execution pipeline.** Both
  `gdr research` (with/without `--plan`) and `gdr plan approve` call it.
  Do NOT duplicate create/poll/render logic. The function accepts
  `previous_interaction_id` and `api_input` for plan-approval cases.
  As of Phase 6 it also accepts derived-domain kwargs
  (`builtin_tools`, `mcp_servers`, `file_search`, `input_parts`,
  `visualization`, `untrusted_input`). The CLI layer parses flags via
  `core/inputs.py` into these types — keep parsing out of
  `execute_research`.
- **`core/inputs.py` is where CLI flags become domain objects.** Every
  parser raises `ConfigError` (never bare `ValueError`), so the CLI's
  try/except → exit code 4 is uniform. `commands/research.py` stays lean
  by delegating all flag parsing here.
- **Auto-untrusted mode fires from `--file` or `--url`** when
  `config.safe_untrusted=True` (the default). `--untrusted-input` also
  trips it manually. The resulting `SecurityPolicy(untrusted=True)` is
  the single place tools get filtered (`code_execution`, `mcp_server`).
  `file_search` is considered safe and stays.
- **Tool ordering in the wire shape is contractual:** builtin simple
  tools → `file_search` → `mcp_server`. Tests assert this explicitly
  (`TestCombinedToolsAndMultimodal`). Reordering changes how dry-runs
  and fixtures look downstream.
- **--mcp-header splits on FIRST `:`** so values like `Bearer abc:123`
  survive. `str.partition("=")` then `str.partition(":")` on the rest.
- **--file-search-store accepts bare names**; the CLI prepends
  `fileSearchStores/`. `FileSearchSpec` still enforces the prefix, so
  config-TOML users get the same guarantee.
- **Auto-append `url_context` tool** when `--url` is used *and* the user
  overrode `--tool`. Handled in `_parse_flag_inputs` via
  `ensure_url_context_tool`. Config defaults already include
  `url_context`, so the branch is only hit when the user narrows tools.
- **`commands/_common.py` is for Phase-7+ family helpers** (client
  build, record lookup, `parse_since`). Do NOT dump research-command
  helpers here — those stay in `commands/research.py`. Rule of thumb:
  if ≥2 command modules need it, factor it to `_common`; otherwise
  keep it local.
- **`parse_since` accepts three forms:** relative (`7d`, `24h`, `30m`,
  `2w`), date-only (`YYYY-MM-DD`), and full ISO 8601 (with `Z` or
  offset). Anything else raises `ConfigError`. The helper takes a `now`
  kwarg for deterministic tests.
- **`gdr resume` never overwrites artifacts by default.** On collision
  it writes to a sibling directory suffixed `_resumed_<ts>`. The user
  opts in to overwriting via `--force`. Record lookup is required — no
  local record means we can't reconstruct a `RunContext`, so the
  command exits 4 and suggests `gdr research`.
- **`gdr cancel` detects missing SDK method gracefully.** Older
  `google-genai` builds may not expose `interactions.cancel`. We check
  via `getattr(client.interactions, "cancel", None)` and print a clear
  "upgrade your SDK" message on absence (exit 4) rather than a raw
  AttributeError.
- **`gdr show` supports id prefix match.** If an exact id lookup misses,
  we try `store.recent()` filtered by prefix. Returns None on ambiguity
  so the user gets the "no record" error (avoids accidental wrong-run
  reads when prefixes collide).
- **`gdr config set` regenerates the TOML file** from the parsed dict.
  Comments are lost. Use `gdr config edit` as the escape hatch for
  comment-preserving or nested changes. This was a deliberate choice
  to avoid adding `tomli-w`/`tomlkit` as a dependency.
- **`gdr config set` validates via Pydantic before writing.** The new
  value goes into the existing dict, and `Config.model_validate` runs
  against the whole thing. If validation fails (unknown key, bad enum
  value, bad type) we refuse to touch the file and exit 4. Means
  users can't get the config into a half-broken state.
- **`gdr config path` uses `typer.echo` (plain stdout), not
  `console.print`.** Rich wraps long paths at terminal width; scripts
  like `path=$(gdr config path)` need the raw string. Same rule applies
  to any command whose output is primarily machine-readable.
- **`gdr doctor` probes network via DNS lookup, not an API call.**
  `socket.gethostbyname("generativelanguage.googleapis.com")` costs
  zero API quota and is the right proxy for "can I reach Google".
  Tests patch `gdr.commands.doctor.socket.gethostbyname` via an
  autouse fixture so the suite never actually hits DNS.
- **`gdr doctor --fix` is idempotent.** Re-running it never re-creates
  existing files/dirs. The test suite exercises this explicitly by
  running `--fix` twice back-to-back and asserting both exits are 0.
- **Attribute-then-key `_get` helper** appears in three modules
  (`rendering.py`, `streaming.py`, `planning.py`). It bridges SDK typed
  objects (attribute access) and dict-shaped test mocks (key access) so
  tests can pass `SimpleNamespace` OR `dict` without a separate adapter.
- **Stream contract:** `interaction.complete` during streaming has
  `outputs=None` per Google's docs. The research command ALWAYS calls
  `.get(id=...)` after the stream ends — whether cleanly or by
  disconnect — to get authoritative outputs. Partial buffers are
  discarded on disconnect. Never reconstruct reports from the stream.
- **MCP header validation is eager.** `build_tools()` validates headers
  BEFORE any wire-shape assembly. One bad header aborts the whole
  request so partial kwargs never reach the API.
- **Untrusted-mode tool filtering runs LAST** in `build_tools()` so the
  `stripped` return list reflects what would actually have been sent.
- **Collaborative planning kwargs are minimal.** `build_plan_kwargs()`
  sets `thinking_summaries="none"` and `visualization="off"` — plans
  should return fast; streaming UI adds no value.

### Rendering
- When printing streamed text, use `console.print(text, end="",
  highlight=False, markup=False, soft_wrap=True)`. Model output may
  contain `[bold]` brackets that would otherwise crash Rich's markup
  parser.
- Text delta buffer in `LiveRenderer` flushes on newline. Use
  `finish()` at end of stream to empty the tail buffer.
- Use Rich's `Status` (not `Live`) for polling spinners — `Live` owns
  the viewport and wipes streamed prints above it.
- **Image outputs are extracted defensively.** `extract_images()` skips
  entries with missing `data` or unparseable base64 so garbage on the
  wire never aborts the full render. The raw transcript still captures
  the original payload for debugging.
- **Image filenames are `image_NNN<.ext>`** (3-digit zero-padded),
  extension from `mimetypes.guess_extension(mime)` with `.png` fallback.
  stdlib may pick `.jpe` for `image/jpeg`; tests check only the prefix.
- **Images section goes *between* body and Sources** in `report.md` —
  tests assert "## Images" before "## Sources" ordering.

### Storage + paths
- Store: `$GDR_STATE_DIR` → `$XDG_STATE_HOME/gdr/` → `~/.local/state/gdr/`.
- Config: `$GDR_CONFIG_PATH` → `$XDG_CONFIG_HOME/gdr/config.toml` →
  `~/.config/gdr/config.toml`.
- Intentionally NOT using `platformdirs` — predictability over platform
  idiom for terminal tooling.
- `env:VAR` in TOML is **prefix-only** (`"env:TOKEN"` works;
  `"Bearer env:TOKEN"` does not). Tested + documented contract.
- Slug sanitization: `[^a-z0-9]+ → -`, max 64 chars, empty falls back to
  `"query"`. id6 fragment (first 6 alnum of interaction id) appended.
  Dashes in id strip out — test IDs in fixtures use dash-free strings
  (`"intabcxyz123"`) so `intabc` is predictable.

### Testing
- Fake the SDK at `google.genai.Client` via `mocker.patch`. Tests pass
  `SimpleNamespace` or `MagicMock` objects; the `_get` helpers accept
  both shapes. See `tests/unit/test_research_command.py::_install_fake_sdk`.
- Golden-file fixtures for streaming live in
  `tests/fixtures/streams/*.jsonl` (one JSON event per line). These
  cover happy path / mid-stream error / out-of-order / disconnect /
  interleaved content.
- Mock `typer.prompt` via `mocker.patch("gdr.core.planning.typer.prompt",
  side_effect=[...])` for interactive-loop tests.
- Rich output capture: `Console(file=io.StringIO(), record=True,
  force_terminal=False, width=120)` then `console.export_text()`.
- All command tests use an autouse `_isolated_env` fixture setting
  `GDR_STATE_DIR` and `GDR_CONFIG_PATH` to tmp_path so tests never
  touch real user state.

### Implementation checklist

#### 1. `.github/workflows/ci.yml`

Create the file. Starting skeleton (works with current `uv.lock`):

```yaml
name: CI

on:
  pull_request:
  push:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  lint-and-test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, macos-latest]
        python-version: ["3.10", "3.11", "3.12", "3.13"]
    steps:
      - uses: actions/checkout@v4
      - name: Install uv
        uses: astral-sh/setup-uv@v3
        with:
          enable-cache: true
      - name: Set up Python ${{ matrix.python-version }}
        run: uv python install ${{ matrix.python-version }}
      - name: Sync deps (frozen to uv.lock)
        run: uv sync --frozen --extra dev
      - name: Ruff lint
        run: uv run ruff check .
      - name: Ruff format check
        run: uv run ruff format --check .
      - name: Mypy (strict)
        run: uv run mypy src
      - name: Pytest with coverage
        run: uv run pytest -q --cov=src/gdr --cov-report=term-missing
```

Notes:
- `astral-sh/setup-uv@v3` handles uv install + cache. Don't use
  `actions/setup-python` separately — `uv python install` after that
  is the more idiomatic flow and avoids Homebrew shadowing on macOS.
- 3.13 is listed; if CI fails on `google-genai` compatibility, drop
  it from the matrix (keep 3.10-3.12). Verify locally first:
  `uv run -p 3.13 pytest -q`.
- `--frozen` requires `uv.lock` to be current. If `pyproject.toml`
  ever gets new deps added, run `uv lock` locally and commit the
  update in the same PR or CI will fail with a lock mismatch.

#### 2. `.github/workflows/release.yml`

Create the file. Skeleton (publishes on `v*` tags, skips `-rc` tags):

```yaml
name: Release

on:
  push:
    tags:
      - "v*"

permissions:
  contents: write   # create GitHub Release
  id-token: write   # required for PyPI OIDC trusted publishing

jobs:
  build-and-publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v3
      - name: Build sdist and wheel
        run: uv build

      # Upload artifacts to the GitHub Release for both RC and
      # stable tags; gives auditors a paper trail.
      - name: Create GitHub Release with artifacts
        uses: softprops/action-gh-release@v2
        with:
          files: dist/*
          draft: false
          prerelease: ${{ contains(github.ref, 'rc') }}

      # PyPI publish ONLY for stable tags (v1.2.3), never for RCs.
      # Requires trusted publisher config on PyPI — see notes below.
      - name: Publish to PyPI
        if: ${{ !contains(github.ref, 'rc') }}
        uses: pypa/gh-action-pypi-publish@release/v1
```

The `if: !contains(github.ref, 'rc')` gate is the thing that lets you
do `v0.1.0-rc.1` as a safe dry-run. RC tags build artifacts and make
a prerelease GitHub Release but do NOT touch PyPI.

### Gotchas to watch for in Phase 10

- **OIDC permissions block.** The release workflow needs both
  `id-token: write` (for PyPI OIDC) AND `contents: write` (for
  GitHub Release creation). Missing either fails with a cryptic
  error like `Resource not accessible by integration`.
- **`uv.lock` must be committed and current.** CI uses
  `uv sync --frozen` so any dep drift fails the build. If a PR
  touches `pyproject.toml`, run `uv lock` in that PR too.
- **Python 3.13 may fail on `google-genai`.** The SDK's minimum
  supported Python shifts over time. If 3.13 goes red in CI but
  3.10-3.12 are green, remove 3.13 from the matrix rather than
  pin-waiting on an SDK release.
- **macOS runners have Homebrew Python.** `actions/setup-python`
  can collide with it silently. Using `astral-sh/setup-uv@v3`
  followed by `uv python install <version>` avoids this — that's
  why the skeleton above is shaped that way. Do NOT use
  `actions/setup-python@v5` in parallel with `uv`.
- **PyPI project name already taken.** Before configuring the
  trusted publisher, check
  https://pypi.org/project/gemini-deep-research/ — if it's taken
  (someone squatted it between now and when you ship), either
  pick a new name (update `pyproject.toml::[project].name`, the
  README install instructions, and the docs) or go through PyPI's
  package-squatting-dispute flow.
- **Pre-release tags must NOT publish to PyPI.** The release
  workflow above gates on `!contains(github.ref, 'rc')`. If you
  ever rename RC tags to something else (`-beta`, `-dev`), update
  that check — otherwise the next beta tag will ship to PyPI.
- **Version drift between `pyproject.toml` and `__init__.py`.**
  Both must match. Before tagging, grep both:
  ```bash
  grep '^version' pyproject.toml
  grep '__version__' src/gdr/__init__.py
  ```
  If they differ, the tag will produce a wheel whose metadata and
  `gdr --version` output disagree. A sync script or a dynamic
  version via `hatchling.version` is a v1.1 improvement.
- **No secret tokens in `dist/*.tar.gz`.** Hatch respects
  `.gitignore` so `progress.txt` and `CLAUDE.md` land in the sdist
  as-is, which is fine (they contain no secrets). But confirm with
  `tar tzf dist/gemini_deep_research-0.1.0.tar.gz | less` before
  the first real PyPI upload.
- **GitHub Actions cache key evolution.** `astral-sh/setup-uv@v3`'s
  `enable-cache: true` keys on `uv.lock` by default. If you
  downgrade to `@v2` the key semantics differ — re-test before
  merging a version bump to the action itself.

### 2026-04-23 - Phase 10 followup: fix CI failures on py3.10
- First CI run on main surfaced 2 job failures (py3.10 x {ubuntu,macos});
  3.11/3.12/3.13 all passed on both OSes. Two distinct bugs:
  1. `src/gdr/commands/config.py:235` used `cast("dict[str, Any]", parsed)`
     on `tomllib.loads()` output. On py3.10 CI runners `tomli` is
     installed (it's a conditional runtime dep
     `python_version < '3.11'`), so mypy sees tomli's real stubs
     returning `dict[str, Any]`, making the cast redundant and
     tripping `--warn-unused-ignores`. On py3.11+ runners `tomli` is
     NOT installed, `ignore_missing_imports = true` kicks in, and the
     cast becomes necessary. Fix: use an inline variable annotation
     `parsed: dict[str, Any] = tomllib.loads(...)` which mypy accepts
     uniformly on both branches, then drop the `cast` import.
  2. `tests/unit/test_models.py:5` imported `from datetime import UTC, datetime`.
     `datetime.UTC` is py3.11+ only. The project's codebase-patterns
     doc (top of progress.txt) explicitly mandates
     `from datetime import timezone` with a module-level
     `_UTC = timezone.utc` alias. The test file was the only violation.
     Fix: apply the project pattern. Place `_UTC = timezone.utc` AFTER
     all imports (ruff E402 would fire otherwise).
- Verified locally on py3.10/3.12/3.13 via
  `uv run -p <ver> --isolated --with-editable . --extra dev <gate>`:
  all 348 tests pass, ruff lint + format clean, mypy strict clean.
- No changes to CI/release workflows or CHANGELOG. Pure source/test
  corrections.
- **Learnings for future iterations:**
  - **mypy's `python_version` setting does NOT match the interpreter's
    conditionally-installed deps.** With
    `[tool.mypy] python_version = "3.10"`, mypy always analyzes the
    3.10 code branch (`import tomli as tomllib`), but *actual* stub
    availability depends on whether tomli is installed in the current
    environment. This makes `cast()` vs annotation choice
    interpreter-dependent. Rule of thumb: prefer inline variable
    annotations over `cast()` at module boundaries where a third-party
    return type is conditionally `Any` vs `dict[...]`. `cast()` is
    fragile under cross-version CI because `warn-unused-ignores` +
    `redundant-cast` will fire asymmetrically.
  - **The codebase pattern "use `_UTC = timezone.utc`" must apply to
    tests as well as source.** The Codebase Patterns section at the
    top of progress.txt implicitly covered tests (it says
    "project-wide"), but the existing `grep` for compliance only ran
    against `src/`. Future phases that add tests should `grep -r
    "datetime import UTC\|datetime.UTC" tests/` before committing.
  - **First push to a fresh remote is the real integration test.**
    Local `uv run` uses whichever interpreter `.python-version` /
    default resolution picks; it does NOT exercise the matrix unless
    you explicitly `-p <ver> --isolated`. Pre-flight for a CI-heavy
    phase should include an explicit 3.10 probe, not just 3.13. In
    Phase 10 I probed 3.13 but not 3.10 - the 3.13 probe passed and
    gave false confidence. Worth adding to the phase checklist.
  - **`gh run view --log-failed`** truncated early (~200 lines) even
    though 8 jobs failed steps had to be fetched. The useful filter
    was
    `gh run view <id> --log-failed | grep -E "error:|Found [0-9]+"`
    to isolate mypy errors across jobs.

### 2026-04-23 - Phase 10 followup 2: RC-aware version guard in release.yml
- First `v0.1.0-rc.1` tag push failed the "Verify version matches tag"
  step: tag `0.1.0-rc.1` compared literally against pyproject version
  `0.1.0` and `__init__.py` version `0.1.0`. The original guard assumed
  tag text exactly equals the PEP 440 package version - true for
  stable tags, false for RC dry-runs where the intent is to ship the
  base version's artifacts to GitHub Release only (no PyPI).
- Fix: strip `-rc.*` suffix from the tag before comparison, so
  `v0.1.0-rc.N` base-matches `0.1.0` but a drift like `v0.2.0-rc.1`
  against pyproject `0.1.0` still fails loudly (which is the intent
  of the guard).
- Verified the shell parameter-expansion logic:
  * `v0.1.0-rc.1` -> base `0.1.0`
  * `v0.1.0`      -> base `0.1.0`
  * `v1.2.3-rc.10` -> base `1.2.3`
- **Learnings for future iterations:**
  - **Test the release workflow with an RC tag before it matters.**
    The version guard was written on the assumption that tag ==
    package version, which is fine for `v0.1.0` but incoherent for
    `v0.1.0-rc.1`. An RC dry-run is cheap and surfaces this kind of
    logic bug before a real release.
  - **PEP 440 and git-tag conventions diverge on prerelease syntax.**
    PEP 440 wants `0.1.0rc1` (no dash, no dot). Git tags conventionally
    use `v0.1.0-rc.1`. If you ever want tag and package version to be
    literally equal, you'd need to either use PEP 440 format in git
    tags (ugly) or derive the package version dynamically from the
    tag (hatchling.version - bigger refactor, deferred to v1.1).
  - **Tag retry workflow after a failed release run:** since no
    artifacts were uploaded and no PyPI publish fired, the failed tag
    can be deleted and re-pushed cleanly:
    ```bash
    git tag -d v0.1.0-rc.1
    git push origin :v0.1.0-rc.1
    # ... after the fix lands on main ...
    git tag v0.1.0-rc.1
    git push origin v0.1.0-rc.1
    ```
    Or sidestep by incrementing the RC number (`v0.1.0-rc.2`). The
    delete+retag approach is cleaner when the prior tag has no
    associated GitHub Release.
  - **Release-workflow step ordering matters.** The version-match
    guard runs BEFORE `uv build` in release.yml, so a guard failure
    costs nothing (no artifacts produced, no PyPI call attempted).
    Keep the guard first - it's the cheapest failure mode.

### 2026-04-23 - Phase 10 followup 3: release creation idempotency + asset glob
- `v0.1.0-rc.1` retag workflow run (24839301960) failed at "Create
  GitHub Release with artifacts":
  ```
  error finalizing release: HttpError: Validation Failed:
  {"resource":"Release","code":"already_exists","field":"tag_name"}
  ```
  Two bugs in a trench coat:
  1. `softprops/action-gh-release@v2` is not idempotent under retag.
     The first RC run on `e124e84` created a release. Retagging
     `v0.1.0-rc.1` to `3d115e3` (after the CHANGELOG commit)
     triggered a second run which found the prior release AND
     created a duplicate draft; finalization errored because the
     tag was already associated with a release. GitHub ended up
     with 2 releases for the same tag name.
  2. `files: dist/*` uploaded `dist/.gitignore` alongside the sdist
     and wheel. Hatchling writes a 1-byte `dist/.gitignore` stub
     to auto-exclude its own output from git. Harmless at PyPI time
     (PyPI publish uses the files in dist/ directly, not the ones
     attached to the Release) but ugly on the Releases page.
- Fix: replace `softprops/action-gh-release@v2` with an explicit
  `gh release create` invocation, preceded by
  `gh release delete "$TAG" --yes --cleanup-tag=false 2>/dev/null ||
  true` so that any prior release for the tag is cleaned before a
  fresh one is created. Narrow the asset list to `dist/*.whl` +
  `dist/*.tar.gz`. Also unified the RC-detection pattern to `-rc.`
  in both the release-creation and PyPI-skip checks (was `rc`
  bare-substring on the PyPI side, which was liberal enough to
  accidentally match words like "archive" in a future tag name).
- **Learnings for future iterations:**
  - **Retagging under softprops/action-gh-release is a foot-gun.**
    The action treats an existing release for the same tag as an
    update target, but racy draft handling during retag corrupts
    state. `gh release create` with an explicit pre-delete is
    simpler and fails loudly in the clean cases. Worth the extra
    ~10 lines of bash.
  - **`dist/.gitignore` is a hatchling artifact.** Any workflow
    uploading `dist/*` to external destinations (Release assets,
    S3, CDN) should use explicit file patterns. It is not uploaded
    to PyPI because `uv build` / twine know to ignore non-distribution
    files.
  - **`contains(github.ref, 'rc')` is too loose.** The ref
    `refs/tags/v0.1.0-rc.1` happens to contain 'rc', but so would
    `refs/tags/v1.0.0-archive`, `refs/tags/v1.0.0-verified`, etc.
    Use `'-rc.'` (with delimiters) to match only the intended RC
    tag format. This applies anywhere github.ref is substring-tested.
  - **Broken-release recovery procedure** (useful reference):
    ```bash
    # Find the stale releases
    gh api repos/OWNER/REPO/releases --jq \
      '.[] | select(.tag_name=="v0.1.0-rc.1") | {id, draft, prerelease}'
    # Delete each by ID (or by tag if unique)
    gh release delete v0.1.0-rc.1 --yes --cleanup-tag=false
    # Delete the tag locally and remotely, then re-push at HEAD
    git tag -d v0.1.0-rc.1
    git push origin :v0.1.0-rc.1
    git tag v0.1.0-rc.1
    git push origin v0.1.0-rc.1
    ```
    Whenever multiple releases collide on one tag name (rare outside
    retag scenarios), delete by explicit release ID rather than
    relying on tag-name-based selection.

    (b) Catch the 400 and print a more actionable error:

        `gdr research --plan` + `gdr plan refine` for the

    (c) File an issue upstream (google-genai / Gemini) asking whether

  - **Where:** `src/gdr/core/client.py:82` — `if not hasattr(client,

    module, which emits the `UserWarning` before `hasattr` returns.
  - **What users see:**

    on every invocation of `status`, `cancel`, `resume`, `follow-up`,

            message=r"Interactions usage is experimental.*",
            category=UserWarning,

    Or filter at module import time in `gdr/cli.py`.

## 2026-04-23 — Versioning + release flow cheat sheet

Written down because this project's release machinery is entirely
manual + tag-driven, and nothing auto-bumps. The next person (or
future-you) will forget the exact sequence.

### Mental model

Manual versioning, tag-triggered release. Two things must happen for
a release to fire:
  1. Bump the literal version string in BOTH `pyproject.toml` (line
     ~7, `version = "X.Y.Z"`) and `src/gdr/__init__.py` (the
     `__version__ = "X.Y.Z"` line).
  2. Push a git tag of shape `v<that-same-version>`.

The tag push is what triggers `.github/workflows/release.yml`
(`on: push: tags: "v*"`). Merging to `main` does NOT release anything.

### Does cutting a version "slate" all subsequent commits for the next release?

By convention yes — but nothing mechanical enforces it. After
`v0.1.0` ships, every commit on `main` is implicitly "what will go
into the next release." The release only happens when you manually
bump + tag.

The "what's queued for next release" list lives in `CHANGELOG.md` as
the `## [Unreleased]` scratch section at the top. Every PR that ships
a user-facing change should add a bullet there. When you're ready to
release, that section gets renamed into the new version header.

### Two tag flavors this project supports

`release.yml` treats them differently (see lines 28-83):

| Tag shape         | GitHub Release         | PyPI publish                |
|-------------------|------------------------|-----------------------------|
| `v0.1.0-rc.1`     | Yes, marked prerelease | **No** (gated at line 81)   |
| `v0.1.0` (stable) | Yes, stable            | **Yes** (OIDC trusted pub)  |

The version-matching guard at line 39 strips the `-rc.N` suffix
before comparing, so `v0.1.0-rc.1`, `v0.1.0-rc.2`, and `v0.1.0` can
all share `version = "0.1.0"` in `pyproject.toml`. That's deliberate:
iterate on RCs at the same base version, and when you cut stable you
don't need another bump.

The PyPI gate at line 81 is `if: ${{ !contains(github.ref, '-rc.') }}`.
The `-rc.` (with both delimiters) is narrow on purpose — see the
earlier entry re: `contains(github.ref, 'rc')` being too loose.

### Typical release flow, start to finish

```bash
# After some PRs have landed with Unreleased CHANGELOG entries:
$EDITOR CHANGELOG.md                          # Promote [Unreleased] -> [X.Y.Z] - <today>
                                              # Add a fresh empty [Unreleased] above it.
                                              # Update compare/tag URLs at the bottom.
$EDITOR pyproject.toml src/gdr/__init__.py    # Bump version in BOTH places.
git commit -am "chore: release X.Y.Z"
git push origin main

# Optional: cut an RC first, smoke-test, then stable.
git tag vX.Y.Z-rc.1
git push origin vX.Y.Z-rc.1                   # -> GH prerelease, NO PyPI
# (install via `pipx install --pip-args='--pre' gemini-deep-research`,
#  verify it works, then proceed.)

git tag vX.Y.Z
git push origin vX.Y.Z                        # -> GH stable release + PyPI publish
```

### SemVer decisions (CHANGELOG declares strict SemVer)

  - Bug fix only -> **patch**: `0.1.0` -> `0.1.1`
  - New feature, backward-compatible -> **minor**: `0.1.0` -> `0.2.0`
  - Breaking CLI/config change -> **major**: `1.0.0` -> `2.0.0`

While < 1.0 the convention is looser: minor bumps can contain
breaking changes. Many projects use `0.x.y` as the "we reserve the
right to break things" signal. This project is at `0.1.0` right now.

### Three places the version string lives — all MUST match

  1. `pyproject.toml` line ~7: `version = "0.1.0"`
  2. `src/gdr/__init__.py`:    `__version__ = "0.1.0"`
  3. Git tag:                   `v0.1.0` (or `v0.1.0-rc.N`)

The CI job "Verify version matches tag" (release.yml line 28-42)
enforces this and fails fast on mismatch. If you ever see that
failure, one of the three drifted — usually because you bumped
pyproject and forgot `__init__.py` (or vice versa).

### If something goes wrong after a tag push

Delete-and-retag is supported (the release job cleans up prior
GitHub releases before creating a new one — see lines 65-68 of
release.yml). The broken-release recovery procedure is documented
in the earlier "ci release hardening" entry above. PyPI is
append-only though; a bad stable upload can only be yanked, not
overwritten. Hence the RC-first habit for anything risky.

  - To ship stable: `git tag v0.1.0 && git push origin v0.1.0`.
